HelloOgre3D 架构流程图

这份文档面向新接手项目的人，用来快速理解 HelloOgre3D 的运行链路、目录职责、AI 框架、脚本桥接和构建流程。 项目当前阶段以 AI 学习与实验沙盒（C++ / Lua）推进：通过一组可运行、可观察的 sample，系统实现并验证游戏 AI 的核心能力——FSM / 决策树 / 行为树、黑板、感知、记忆、知识源、团队通信、影响力地图与战术层；长期目标是演进为生产级游戏项目 / 生产级玩法运行时。 下文图中重点标出“应用编排、运行时适配、沙盒玩法、AI 分层架构、脚本扩展、资源加载”之间的边界。方向定调以 docs/project-direction.md 为准。

当前主循环的关键点是：渲染帧由 Ogre 驱动，游戏逻辑从 ClientManager 进入，再传递到 GameManager 和沙盒对象系统。 ObjectManager::Update 现在除了驱动对象更新，还集中承担 AIScheduler 分帧错峰、空间索引重建、TeamBlackboard 分批更新和事件队列 flush——职责偏重，是后续把跨对象逻辑 system 化的重点对象。对象数量增加时，最先要关注的是空间查询、感知扫描、Lua 高频调用、AI tick 和渲染侧成本。

AI 已从单纯的 FSM / 决策树 / 行为树，扩展为一套分层结构（对应 Chapter 7-9 学习主线）：

  • 感知层 ai/perception：VisionSensor 视野扫描 + MemoryStore 最后已知位置（带 TTL / confidence 衰减）；AgentSpatialIndexSystem 提供 uniform grid 空间查询；AgentPerceptionQuery 统一找敌 / 可达判定，结果写回 Blackboard。
  • 决策层：AIController（组件）持有 FSM / DecisionTree / BehaviorTree 三类 driver，共享同一个 Blackboard（已支持 object-id / 数组 / TTL / confidence / metadata）。行为树支持 Parallel / Random 节点与 Selector / Sequence 的反应式重评估，结构走 Lua 声明式配置。
  • 协调层：TeamBlackboardService 共享队伍敌情；InfluenceMapSystem 提供危险 / 支援层做战术移动；SandboxEventDispatcher 提供 Local / Team / Global 事件分发；AIScheduler 做分帧错峰。
  • 依赖注入：core/SandboxServices 向组件 / 系统提供运行时服务（对象查询、物理、寻路、脚本），逐步替代 g_* 全局单例。

样例映射：Sandbox6-8 覆盖 FSM / DT / BT；Sandbox9 Chapter 7 Knowledge；Sandbox10-11 / 14 Chapter 8 Perception / Hearing；Sandbox12 TeamBlackboard；Sandbox13 InfluenceMap；Sandbox15 Chapter 9 Formation。

四、目录职责

src/HelloOgre3D/client

程序入口和 Ogre 帧回调层。主要负责启动应用、接收 frameStarted / frameRenderingQueued，并把控制权交给游戏侧 manager。

src/HelloOgre3D/game

应用编排层。负责窗口、场景、输入、样例生命周期、Lua 绑定入口和主要 manager 的连接。

src/HelloOgre3D/runtime

运行时适配层。承接 Ogre、Tracy 等和引擎或工具强相关，但又不属于纯玩法规则的能力。

src/HelloOgre3D/common

通用基础设施。包含 Lua VM、日志、文件工具、基础类型等跨模块能力。

src/HelloOgre3D/sandbox

沙盒玩法与 AI 层，是当前实验主战场。AI 子模块：ai/behavior（行为树）、ai/decision（决策树）、ai/fsm（状态机）、ai/perception（视觉/记忆/空间查询）、ai/team（团队黑板）、ai/tactics（影响力地图）、ai/common（Blackboard / AIScheduler）、ai/navigation。另含 objects、components（含 components/ai/AIController）、systems、core/SandboxServices（依赖注入）、core/event（事件总线）、scripting（脚本桥）。

bin/res/scripts

Lua 运行脚本。包含基础库、manager、agent、AI 配置和 samples，是验证玩法行为的重要入口。

七、当前演进重点

当前阶段以 AI 学习与实验沙盒为主线，长期目标是生产级项目：Chapter 7-9（Knowledge / Perception / Tactics）已基本落地，下面是已完成面与下一步重点。详见 docs/high-priority-todo.md、docs/AIArchitectureBeyondBook.md。

AI 分层（已落地）

  • 感知：VisionSensor / MemoryStore / AgentPerceptionQuery / 空间索引。
  • 黑板：object-id、数组、TTL、confidence、metadata。
  • 团队 / 战术：TeamBlackboardService、InfluenceMapSystem、Formation。

行为树补强（已落地）

  • Parallel / Random 节点 + Lua 声明式配置。
  • Selector / Sequence 反应式重评估（条件变化即中断）。
  • BehaviorTrace + 黑板摘要 + 配置错误诊断。

下一步重点

  • uniform grid 空间分区替换线性查询（替换点已留）。
  • 把 TeamBlackboard / 空间索引等从 ObjectManager 拆成独立 system。
  • 继续以 SandboxServices 收口 g_*，组件所有权迁 unique_ptr。

已知待修（code review）

  • Blackboard SetAgent 存裸指针，改存 object-id 防悬挂。
  • Random 节点用全局 rand()，破坏可复现，应走可控种子。
  • 反应式重评估对非幂等 Action 会重入，前缀宜限纯条件。

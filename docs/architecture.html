二、主循环流程

当前主循环的关键点是：渲染帧由 Ogre 驱动，游戏逻辑从 ClientManager 进入， 再传递到 GameManager 和沙盒对象系统。对象数量增加时，最先需要关注的是对象线性更新、Lua 高频调用、AI tick、动画和渲染侧阴影成本。

AI 体系目前同时包含有限状态机、决策树和行为树。行为树结构已经开始向 Lua 配置迁移， C++ 侧负责运行时节点、tick 调度和宿主对象桥接，Lua 侧负责样例规则和可扩展行为。

四、目录职责

src/HelloOgre3D/client

程序入口和 Ogre 帧回调层。主要负责启动应用、接收 frameStarted / frameRenderingQueued，并把控制权交给游戏侧 manager。

src/HelloOgre3D/game

应用编排层。负责窗口、场景、输入、样例生命周期、Lua 绑定入口和主要 manager 的连接。

src/HelloOgre3D/runtime

运行时适配层。承接 Ogre、Tracy 等和引擎或工具强相关，但又不属于纯玩法规则的能力。

src/HelloOgre3D/common

通用基础设施。包含 Lua VM、日志、文件工具、基础类型等跨模块能力。

src/HelloOgre3D/sandbox

沙盒玩法层。包含对象系统、AI、导航、脚本桥、物理、组件和系统，是当前重构与功能实验的主要区域。

bin/res/scripts

Lua 运行脚本。包含基础库、manager、agent、AI 配置和 samples，是验证玩法行为的重要入口。

七、当前演进重点

对象系统降耦

  • 缩短对象继承链。
  • 把动画、AI、物理、武器挂接逐步拆成组件。
  • 减少 manager 和对象之间的隐式全局依赖。

AI 可配置化

  • 行为树结构从 C++ / Lua builder 迁移到 Lua 配置。
  • 保留 C++ 节点执行效率，同时让 Lua 承接玩法规则。
  • 后续补黑板、触发器和节点级调试统计。

性能可观测

  • Tracy 已接入，用于查看帧、Lua、AI、对象更新热点。
  • 下一步应补对象数、Lua 调用数、BT tick 数、渲染统计等 counters。
  • 对象增多后的瓶颈要同时看逻辑侧和渲染侧。

跨平台整理

  • Windows 仍是主要路径，同时已开始补 macOS / Xcode 支持。
  • 新增路径和 include 需要注意大小写敏感。
  • 引擎耦合逻辑优先收口到 runtime。

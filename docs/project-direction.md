# 项目方向总纲（北极星）

> 这份文档是 `HelloOgre3D` 的**最高层方向定调**，统领 `docs/` 下其余路线/方案/分析文档。做任何取舍前先读这里。
>
> 它回答三个问题：**项目要变成什么 / 核心主轴是什么 / 按什么顺序补哪些模块**。细节在各专项文档（见 §7 索引），本文只给定调、整合优先级和第一个垂直切片。

---

## 1. 项目身份（已确认）

**从“教学 sample 沙盒”切换为“数据驱动的沙盒玩法运行时 + 编辑器”。**

- 旧身份：通过重构游戏案例沉淀开发经验，产出一批可运行的 chapter / sandbox sample，内容（对象、AI、玩法）都是手写代码。
- 新身份：一个**数据驱动**的沙盒平台——大世界、多生物、可配置 AI、生物触发器、UGC 玩法；内容是**可作者编辑、可序列化、可热重载的数据**，而不是代码。
- sample 不废弃，但定位变成“运行时与编辑器的验证场 / 回归面”，不再是终点。

> 这一身份切换是后续所有优先级的判断基准：**凡是把“内容代码化”继续往前推的，降优先级；凡是把“内容数据化”地基补上的，升优先级。**

## 2. 核心主轴：内容即数据

对标生产级沙盒 MiniGame（详见 `docs/reference-minigame-patterns.md`）后的最重要结论：

> **大世界多生物、生物触发器、AI 可视化配置、UGC —— 这四个目标全部压在同一条脊梁上：把内容从代码变成数据（定义 + 序列化 + 运行时实例化 + 热重载）。**

它们是同一套基建的不同出口：

| 目标 | 它真正需要的底层 |
|---|---|
| 多种生物不靠抄代码 | 生物 = 组件组合 + `Def` 配置数据（CSV/Lua 表原型） |
| 生物带触发器 | 统一事件总线 + ECA + 参数化事件名（`事件?objid=N`） |
| AI 可视化配置 | 数据驱动行为树（已有加载器）+ 可视化编辑器（FGUI 积木） |
| UGC 玩法 | 实体/行为/规则的可序列化 schema + 运行时实例化 + 沙箱 |

**HelloOgre3D 当前这条脊梁基本为 0**：无 CSV/数据驱动 Def 体系、无生产级跨对象事件总线（仅有同步 dispatcher 骨架）、无触发器系统、对象仍是 C++ 继承 + 全局单例耦合。这是和目标之间最大的鸿沟，也是接下来的主战场。

## 3. 模块优先级总图（L0–L4）

把各专项文档的缺口拉通成一张按“建造层”排的图。**铁律：地基阻塞一切，越往上越能等。**

```
L4  UGC平台        │ ⑫ 内容打包/加载/沙箱/Mod式扩展(拷贝原型+ID段隔离+进退图还原)
                   │        ▲ 前面不稳不能碰
L3  工具/编辑器     │ ⑨ 通用积木编辑器框架(FGUI) → ⑩ AI编辑器 → ⑪ 触发器编辑器
                   │        ▲ FGUI 重投入在此兑现价值
L2  规模(大世界)    │ ⑦ 空间分区+AI LOD+感知原语库   ⑧ 世界分块/streaming(看大世界规模再定)
                   │        ▲ 多生物必须，否则全场线性扫描必炸
L1  数据驱动玩法    │ ④ 生物数据化(组件组合,零C++子类)
                   │ ⑤ BT补强: G4(Parallel/Random)→G3(参数运行时求值)→G1(中断/重评估)
                   │ ⑥ 触发器运行时: 事件源(区域/碰撞/计时)+ECA(手写Lua+沙盒)+Trigger.X API+事件喂BT
L0  地基(先做)     │ ① 解耦: SandboxServices注入, 去全局单例/反向依赖
                   │ ② CSV/数据驱动 Def 体系(只读原型+持指针+就地替换热重载)  ← 脊梁的根
                   │ ③ 统一事件总线(四件套+参数化事件名+Local/Team/Global)
```

## 4. 第一个垂直切片（最关键的执行建议）

诚实判断：这个目标的 scope **比 MiniGame 一个团队做的还大**，而这是个人/业余项目。**按 12 个模块平铺推进 = 地基永远修不完、永远看不到玩法。** 所以：

> **先定义并打通一个最小垂直切片，用它把整条数据脊梁验证一遍，而不是逐层修完。**

### 切片定义

> **一种用数据定义的生物，放进世界，带一个区域触发器（玩家进入 → 生物攻击），AI 走数据驱动行为树，全部能热重载。全程手写数据，不做任何编辑器。**

### 这一个切片强制串起的能力

| 能力 | 来自 |
|---|---|
| `Def` 体系 + 运行时实例化 | L0-② |
| 组件装配（生物=组件组合） | L1-④ |
| 统一事件总线 + 参数化事件名 | L0-③ |
| 区域事件源（TriggerVolume） | L1-⑥ |
| ECA 触发器运行时（手写 Lua + 沙盒） | L1-⑥ |
| 数据驱动行为树（已有加载器）+ 事件喂 BT | L1-⑤ |
| 去全局单例（context 注入贯穿） | L0-① |

### 为什么是它

- 编辑器只是“给这套数据加的 GUI 前端”——切片跑通后，编辑器是叠加，不是地基（呼应 `visual-editor-implementation-plan.md` 的“先闭环后 GUI”）。
- 打通它 = 证明整条脊梁可行，后面每个能力都在它上面叠；打不通 = 做再多编辑器/UGC 都是空中楼阁。

## 5. 先做 / 缓做 / 别做

| 档位 | 内容 | 理由 |
|---|---|---|
| **先做** | L0 全部（解耦 + Def 体系 + 事件总线）→ 第一个垂直切片 | 地基复利最高，是其他一切的前提 |
| **缓做** | 编辑器(L3)、UGC/Mod(L4)、世界 streaming(L2-⑧) | 等切片跑通、数据格式稳定再上 |
| **别做 / 维持** | FairyGUI 封版停加控件；不给运行时玩法对象上反射；macOS 迁移搁置 | 已过价值拐点 / 两轨教训 / 非硬需求 |

两条容易反复踩的判断：
- **FGUI 已够用**，停止加新控件，它的价值在“当编辑器 UI 基座”，不在更多业务控件。
- **反射只给编辑器节点，不给运行时生物**（MiniGame 两轨教训，见 `reference-minigame-patterns.md` §1）：运行时玩法对象走 CSV+轻量序列化+组件，反射+flag 多通道留给编辑器层。

## 6. 高层节奏

> 不设日期（业余项目，按可用时间推进）；只定顺序与“完成定义”。

1. **地基期**：L0-①解耦 + L0-②Def 体系 + L0-③事件总线。完成定义：生物能由数据实例化、组件不再依赖全局单例、事件能按名注册/分发。
2. **切片期**：打通 §4 垂直切片。完成定义：手写一份生物 Def + 一个区域触发器 + 一棵数据驱动 BT，进游戏后“玩家靠近→生物攻击”成立，改数据热重载生效。
3. **数据驱动玩法期**：补 BT（G4/G3/G1）、触发器事件源扩展、再加几种数据定义的生物。完成定义：新增一种生物/一条触发逻辑只改数据。
4. **规模期**：空间分区 + AI LOD + 感知原语库（多生物不卡）。
5. **编辑器期**：FGUI 积木框架 → AI 编辑器 → 触发器编辑器（给前面的数据加 GUI 前端 + 运行态调试回传）。
6. **UGC 期**：内容打包/加载/沙箱/Mod 式扩展。

## 8. 风险与心法

- **最大风险是 scope 失控**：目标比团队级项目还大，业余时间有限。对策就是 §4——**始终用一个垂直切片牵引，先闭环再铺面**，拒绝平铺式修地基。
- **复利思维**：L0 地基每修好一块，上面所有模块都变便宜；反之在耦合/硬编码的地基上加功能，每次都更贵。这是“先做地基”的根本理由。
- **不照搬历史包袱**：MiniGame 的因子系统、四层并存、dev 子 VM、FlatBuffer、ID 段重叠都是历史债，新项目直接用更干净的一套（见各 gap 文档的“不照搬”小节）。
